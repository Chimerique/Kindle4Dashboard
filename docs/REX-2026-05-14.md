# Retour d'expérience - Kindle Dashboard HA

Date de référence : 2026-05-14

Ce document regroupe les points opérationnels les plus utiles observés pendant le déploiement et le dépannage du Kindle Dashboard HA. L'objectif est de conserver ici les incidents rencontrés, les correctifs appliqués et les vérifications à refaire en cas de réinstallation.

## Résumé des problèmes résolus

- La Kindle pouvait entrer en veille profonde / RTSuspend pendant les longues attentes du scheduler, ce qui figeait la boucle de mise à jour.
- Le mode USBNetwork n'était pas fiable au démarrage tant qu'il n'était pas relancé explicitement.
- La configuration WiFi était incomplète : `wpa_supplicant.conf` ne contenait pas les credentials NoTrespassing, et `ap_scan=2` empêchait la connexion automatique.
- Le flag `-d` (debug) codé en dur dans `/etc/init.d/wpa_supplicant` faisait grossir `/var/log/wpa_supplicant` jusqu'à saturer le disque RAM de 32MB en quelques heures, rendant toute opération réseau impossible (`/tmp` plein, wget impossible).
- `wifid` (daemon WiFi natif Kindle) ne connaît que les réseaux de sa base interne chiffrée (`/var/local/system/wifid.conf`). Il ne connaissait que `Livebox-95A0` et jamais `NoTrespassing` → wpa_supplicant restait en état `INACTIVE` après chaque `lipc wifid enable 1`.
- `update_frame.sh` ne faisait pas de DHCP (`udhcpc`) après la connexion WPA2, utilisait `wget -T 30` (option non supportée par BusyBox 1.7.2), et `ip route` absent sur ce firmware (il faut `route -n`).
- Le dashboard pouvait rester désactivé à cause du flag `DASHBOARD_DISABLED`.

## Correctifs appliqués

- Le scheduler utilise maintenant un sommeil robuste par paliers avec vérification de l'heure réelle.
- La protection écran et la prévention de suspension sont activées ensemble pendant l'affichage du dashboard.
- Une alarme RTC est programmée de manière best-effort avant les longues attentes.
- USBNetwork est relancé automatiquement au démarrage pour garantir un accès SSH de secours.
- `/etc/wpa_supplicant.conf` : credentials NoTrespassing ajoutés (PSK hashé, pas en clair), `ap_scan` passé de 2 à 1.
- `/etc/init.d/wpa_supplicant` : flag `-d` supprimé. Un subprocess background (délai 7s) injecte NoTrespassing dans wpa_supplicant via wpa_cli, après le `syncNetworkList` de wifid, rendant le WiFi fonctionnel aussi depuis l'UI native Kindle.
- `update_frame.sh` – `enable_wifi()` : après `lipc wifid enable 1` (hardware ON), redémarre wpa_supplicant via `/etc/init.d/wpa_supplicant`. wifid ne connaît pas NoTrespassing dans sa base interne chiffrée, ce redémarrage est indispensable pour les cycles automatiques.
- `update_frame.sh` – `wait_for_ip()` : appelle `udhcpc` dès que `wpa_state=COMPLETED` pour obtenir l'IP et la route par défaut.
- `update_frame.sh` – `download_image()` : `wget -T 30` → `wget` sans `-T` (non supporté par BusyBox 1.7.2).
- `update_frame.sh` – `ensure_gateway()` : `ip route` → `route -n` (commande `ip` absente sur ce firmware).

## Architecture WiFi (Kindle 4 NT)

wifid (daemon natif) maintient sa propre base chiffrée `/var/local/system/wifid.conf` qui ne contient que les réseaux enregistrés via l'UI ou WPS. wifid démarre wpa_supplicant via init.d, puis appelle `syncNetworkList` qui **écrase** la liste des réseaux avec les siens. Si NoTrespassing n'est pas dans la base wifid, wpa_supplicant se retrouve avec uniquement Livebox (DISABLED) → état INACTIVE, pas de connexion.

Fix : 7 secondes après le démarrage de wpa_supplicant (après le sync wifid), un subprocess background injecte NoTrespassing avec le PSK hashé via wpa_cli et déclenche le DHCP après la connexion WPA2.

## Points importants à retenir

- Le fichier `DASHBOARD_DISABLED` dans `/mnt/us/` désactive le dashboard.
- Le WiFi est activé pendant les mises à jour puis coupé ensuite pour économiser la batterie.
- Si le réseau USB n'est pas disponible, le WiFi doit servir de voie de secours une fois connecté.
- Le Kindle 4 peut rester stable avec une image e-ink affichée via `eips` sans refresh logiciel continu.

## Vérifications utiles après redéploiement

1. Vérifier que le scheduler tourne et journalise des cycles réguliers.
2. Vérifier que `preventSuspend` et `preventScreenSaver` sont bien rétablis à la fin d'un cycle.
3. Vérifier que USBNetwork expose bien l'accès SSH de secours.
4. Vérifier que `wpa_supplicant` se reconnecte après un redémarrage réseau.
5. Vérifier que le dashboard affiche bien l'image Home Assistant après suppression du flag de désactivation.

## Commandes de diagnostic utiles

- `tail -20 /tmp/scheduler.log`
- `wpa_cli -i wlan0 status`
- `ifconfig wlan0`
- `ls /mnt/us/DASHBOARD_DISABLED`

## Éléments à surveiller

- Si le Kindle cesse de rafraîchir l'écran, vérifier d'abord la suspension système avant de suspecter Home Assistant.
- Si le WiFi reste bloqué en handshake, redémarrer `wpa_supplicant` après avoir vérifié la configuration.
- Si l'accès SSH disparaît, confirmer que USBNetwork est toujours lancé et que la configuration USB/RNDIS est active.
